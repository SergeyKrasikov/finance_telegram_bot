# Delivery And Migration Policy

## Зачем нужен этот документ

Этот документ фиксирует общий принцип проекта: серьёзные изменения нельзя выкатывать напрямую в прод. Для финансового продукта недостаточно только "новый код работает". Нужно отдельно доказать, что:

- тестовая сборка проходит;
- миграция данных безопасна;
- пользовательский контракт не ломается;
- есть понятный порядок перехода с продовой версии на новую.

## Базовый принцип

Любое заметное изменение должно проходить путь:

1. проектное решение;
2. реализация;
3. полная тестовая сборка на тестовой ветке;
4. проверка миграции;
5. прогон валидаций;
6. только после этого rollout в прод.

## Текущий формат test rollout

На текущем этапе тестовой сборкой считается:

- полная сборка на тестовой ветке;
- прогон тестов на этой ветке;
- проверка, что изменения готовы к дальнейшему migration rollout.

## Что считается серьёзным изменением

- изменения financial core logic;
- изменения monthly и cascade logic;
- изменения истории, ledger, cash flow и backfill;
- изменения exchange logic и расчёта стоимости активов;
- изменения структуры графа распределения;
- изменения, влияющие на Mini App отчёты, балансы и пользовательские цифры;
- любые schema changes;
- любые изменения, которые требуют адаптации старых данных под новую модель.

## Обязательные артефакты для серьёзного изменения

Для каждого серьёзного изменения должны существовать:

- краткое описание цели;
- список затронутых инвариантов;
- тестовый сценарий;
- migration plan;
- rollout plan;
- список post-deploy проверок;
- описание rollback или безопасной остановки rollout.

## Стандартный шаблон migration plan

### 1. Что меняется

- какая продуктовая или техническая возможность добавляется;
- какие таблицы, функции, данные или контракты затрагиваются;
- что должно остаться совместимым.

### 2. Риски

- что может сломать историю;
- что может сломать monthly;
- что может сломать balance/report contract;
- какие места особенно чувствительны для prod data.

### 3. Подготовка

- какие schema changes нужно применить;
- нужен ли backfill;
- нужен ли seed;
- нужно ли dual-write или compatibility layer;
- какие feature flags или runtime toggles нужны, если они есть.

### 4. Test rollout

- как поднимается и обновляется тестовая ветка;
- какие фикстуры или копия данных нужны;
- какие проверки обязательны до допуска в прод;
- какие пользовательские сценарии должны быть проверены руками.

### 5. Prod rollout

- порядок выката;
- шаги включения;
- момент переключения runtime;
- кто и что проверяет сразу после включения.

### 6. Post-deploy validation

- сверка balances;
- сверка history;
- сверка monthly;
- сверка exchange;
- сверка ключевых экранов и отчётов Mini App;
- проверка text-bot fallback.

### 7. Rollback

- можно ли откатить код;
- можно ли остановить новый runtime;
- какие данные нельзя удалять при откате;
- как диагностировать частично завершённую миграцию.

## Release gate

Изменение не готово к продовому rollout, если:

- нет test build validation;
- нет понятного migration plan;
- не зафиксированы инварианты;
- не определены post-checks;
- нет решения по rollback или хотя бы safe-stop;
- не проверен пользовательский контракт для отчётов, history и balances.

## Особые правила для этого проекта

### Monthly

- любые изменения monthly считаются high-risk;
- каскадный отчёт проверяется отдельно;
- нельзя менять monthly только на основании "код стал чище";
- до продового rollout нужен отдельный checklist.

### History

- история должна сохранять смысл и связность;
- перенос на новую структуру не должен скрывать старые операции;
- при необходимости должны сохраняться связи legacy -> new model.

### Encryption

- изменения в модели шифрования считаются high-risk;
- per-user key management должен проверяться отдельно;
- rollout шифрования должен иметь явный план backfill/migration;
- ключи пользователей не должны оказываться в базе или логах;
- до prod rollout нужно отдельно проверить поведение при отсутствии ключа, ошибке расшифровки и частично мигрированных данных.

### Exchange

- обмены должны оставаться аудируемыми;
- должны сохраняться количество, цена и направление обмена;
- миграция не должна ломать отображение капитала в базовой валюте.

### Mini App и text bot

- новый UI не должен ломать цифры;
- text bot fallback должен оставаться рабочим после rollout;
- тестовый прогон должен покрывать и основной UI, и базовые bot-сценарии.
